💡
Idea clave: CLAUDE.md es la memoria persistente de tu proyecto. Todo lo que escribís ahí, Claude lo sabe desde el primer mensaje de cada conversación — sin que tengas que repetirlo.

📁 Ubicaciones y diferencias

UbicaciónDiferenciaCuándo cargaAlcance
./CLAUDE.md En la raíz del proyecto, visible en el repositorio Siempre que Claude corra en ese directorio Todo el proyecto
./.claude/CLAUDE.md Dentro de la carpeta .claude/. Equivalente a ./CLAUDE.md en alcance. Útil para mantener la raíz limpia. Siempre que Claude corra en ese directorio Todo el proyecto
~/.claude/CLAUDE.md CLAUDE.md personal global. Aplica a todos tus proyectos. Para tus preferencias personales de trabajo. Siempre, en cualquier proyecto Todos tus proyectos
./src/CLAUDE.md Instrucciones específicas para un subdirectorio. Solo aplica cuando Claude trabaja dentro de src/. On-demand: solo cuando Claude trabaja dentro de ese directorio Solo ese subdirectorio
📌
¿CLAUDE.md vs .claude/CLAUDE.md? Son equivalentes en alcance — ambos cargan siempre que Claude corra desde ese directorio. La diferencia es de visibilidad: CLAUDE.md en la raíz es más visible para el equipo; .claude/CLAUDE.md agrupa toda la configuración de Claude en un lugar. Elegí según la convención del proyecto.
📌
Carga aditiva por jerarquía: Claude combina todos los CLAUDE.md relevantes en cascada — global → proyecto → subdirectorio. Los más específicos tienen precedencia si hay contradicciones. Los de subdirectorios cargan on-demand.

✍️ ¿Qué poner en CLAUDE.md?

  • 🏗️ Arquitectura del proyecto — estructura de capas, patrones usados, decisiones importantes
  • 📐 Estándares de código — estilo, naming conventions, linters, formateadores
  • 🚫 Restricciones explícitas — "nunca modificar migraciones", "no usar DbContext en controllers"
  • 🛠️ Stack y comandos frecuentes — build, test, run, deploy
  • 🔗 Contexto de negocio — glosario del dominio, entidades clave

📏 Longitud recomendada

CLAUDE.md se carga en cada conversación y consume tokens del contexto. Mantenelo bajo las 200 líneas. El material extenso va mejor en skills con progressive disclosure:

💡 Ejemplo real completo

# EcommerceApi

## Stack
- .NET 8 · ASP.NET Core Minimal APIs
- Entity Framework Core 8 · SQL Server 2022
- xUnit · Moq · FluentAssertions

## Arquitectura
Clean Architecture: Domain → Application → Infrastructure → API
- Commands y Queries separados (CQRS)
- Repositorios abstraen EF Core de Application

## Convenciones de código
- Record types para DTOs y Value Objects
- Métodos async terminan en Async
- Nunca retornar null: usar Result<T> del paquete FluentResults
- EF Core: AsNoTracking() en queries de solo lectura

## Restricciones
- Nunca modificar migraciones existentes — crear nueva migración
- No usar DbContext en Controllers ni Application — solo en Repositories
- No hardcodear connection strings

## Comandos
- build:   `dotnet build`
- test:    `dotnet test --filter Category=Unit`
- lint:    `dotnet format`
- migrate: `dotnet ef migrations add <Nombre>`

🚀 Generar uno automáticamente con /init

Si el proyecto ya tiene código, Claude puede analizar el repositorio y generar un CLAUDE.md inicial:

/init

Claude lee la estructura del proyecto, los archivos de configuración (package.json, .csproj, pyproject.toml, etc.) y propone un CLAUDE.md con el stack y convenciones detectadas. Revisá y editá el resultado antes de commitearlo — Claude puede detectar el stack correctamente pero no conoce tus restricciones de negocio.

⚠️
Una instrucción en el chat puede prevalecer sobre CLAUDE.md si contradice directamente lo que dice. CLAUDE.md establece el comportamiento por defecto, no reglas absolutas. Para restricciones que deben ser imposibles de saltear, usá permisos en settings.json (deny rules).
