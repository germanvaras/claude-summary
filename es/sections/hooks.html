🪝 Hooks Automatización↗ Docs

💡
Idea clave: Los hooks son automatizaciones event-driven — se disparan solos cuando Claude hace algo, sin que vos lo pidás. Configurás una vez, funciona siempre.

🔔 Eventos disponibles

EventoCuándo se disparaCaso de uso típico
PreToolUseJusto antes de que Claude use una herramientaValidar parámetros, bloquear comandos peligrosos, logging de auditoría
PostToolUseDespués de que Claude usa una herramienta (éxito)Correr linter o build tras escribir/editar, notificaciones
NotificationCuando Claude envía una notificación al usuarioLogging, alertas externas, integración con Slack
StopCuando Claude termina de responderLimpieza, reportes post-tarea, notificaciones de fin
UserPromptSubmitCuando el usuario envía un mensajeEnriquecer el prompt automáticamente, auditoría de inputs

🏗️ Estructura en settings.json

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          { "type": "command", "command": "npm run lint" }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "/ruta/absoluta/validate.sh" }
        ]
      }
    ]
  }
}
📌
El campo matcher es una expresión regular que se compara contra el nombre de la herramienta. El operador | funciona como OR de regex: "Write|Edit" activa el hook en Write o Edit. Omitir matcher hace que el hook dispare ante cualquier herramienta.

📥 Qué información recibe el hook

Claude Code pasa la información del evento al proceso del hook via stdin como JSON. El hook puede leer ese JSON para saber exactamente qué hizo Claude.

Estructura para PreToolUse y PostToolUse:

# stdin que recibe el hook:
{
  "tool_name": "Write",
  "tool_input": {
    "file_path": "/src/OrderService.cs",
    "content": "..."
  }
}

# Para PostToolUse, también incluye la respuesta:
{
  "tool_name": "Write",
  "tool_input": { "file_path": "..." },
  "tool_response": { "success": true }
}

Ejemplo de script de hook que usa esta información:

#!/bin/bash
# validate-before-write.sh
INPUT=$(cat)   # lee el JSON del stdin
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path')

if [[ "$FILE" == *"migrations"* ]]; then
  echo "ERROR: No se pueden modificar archivos de migración directamente."
  exit 2   # exit 2 = BLOQUEAR la acción
fi

exit 0   # exit 0 = PERMITIR la acción

🚦 Cómo PreToolUse puede bloquear acciones

Un hook PreToolUse puede decidir si Claude puede o no ejecutar la herramienta, basándose en su código de salida:

Exit codeComportamientoCuándo usarlo
0Acción permitida, Claude continúa normalmenteValidación exitosa
2Acción bloqueada. Lo que el hook escribió a stdout se le muestra a Claude como explicación del bloqueo.Cuando detectás algo que Claude no debería ejecutar
Otro (≠0, ≠2)Error del hook — Claude recibe una notificación de error pero puede continuarFallo inesperado del script
✅
Lo que el hook escribe a stdout con exit 2 llega a Claude como mensaje de feedback. Claude lo lee y puede ajustar su plan. Escribí mensajes claros: "Archivo de migración protegido — usá el script create-migration.sh".

❌ Qué pasa cuando un hook falla

Tipo de hookSi falla (exit code distinto de 0 y 2)
PreToolUseClaude recibe una notificación de error del hook. La acción puede continuar o detenerse según la configuración.
PostToolUseClaude recibe el output del error y puede reaccionar — por ejemplo, si el linter falla, Claude puede ver los errores e intentar corregirlos.
StopEl error se loguea pero no interrumpe el flujo — el hook de fin no puede deshacer lo que Claude ya hizo.

🛣️ Variables de entorno en hooks

Los hooks corren con el PATH del sistema, no del shell interactivo. Esto causa la mayoría de los "command not found" en hooks. Reglas prácticas:

  • Usá rutas absolutas para ejecutables: /usr/local/bin/npm en vez de npm
  • Si el comando necesita estar en el directorio del proyecto, prefijá con cd /ruta/proyecto &&
  • Los hooks heredan las variables de entorno del proceso de Claude Code, incluyendo las de tu sesión de terminal
  • En Windows, verificá con where npm la ruta completa y usala en el hook
# ❌ Puede fallar — npm no está en el PATH del hook
"command": "npm run lint"

# ✅ Ruta absoluta — siempre funciona
"command": "C:\\Users\\usuario\\AppData\\Roaming\\npm\\npm.cmd run lint"

# ✅ O cambiar al directorio del proyecto primero
"command": "cd C:\\proyecto && npm run lint"

📋 Ejemplos prácticos

Linter automático tras editar archivos TypeScript:

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "npx eslint --fix {tool_input.file_path}" }]
    }]
  }
}

Build automático tras cambios en .NET:

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "dotnet build --no-restore -q 2>&1 | tail -5" }]
    }]
  }
}

Notificación al terminar una tarea larga:

{
  "hooks": {
    "Stop": [{
      "hooks": [{ "type": "command", "command": "powershell -Command \"[System.Media.SystemSounds]::Beep.Play()\"" }]
    }]
  }
}
⚠️
Los hooks corren comandos reales en el sistema. Probálos con claude --verbose en entornos de desarrollo antes de usarlos en producción. Un hook mal escrito en PreToolUse puede bloquear todas las operaciones de Claude.
