🧠 Modelo Mental Por dónde empezar

💡
Antes de meterte en los detalles: entendé qué hace cada pieza. Si esto queda claro, el resto de la documentación va a encajar solo.

🗺️ El mapa completo

PiezaRolCuándo aplica
📄 CLAUDE.mdMemoria persistente del proyectoSiempre — carga en cada conversación automáticamente
🧩 SkillsConocimiento especializado activableCuando la tarea coincide semánticamente con la descripción de la skill
🪝 HooksAutomatizaciones disparadas por eventosCuando Claude hace algo: escribe un archivo, ejecuta un comando, termina
🔌 MCP ServersConexión a herramientas externasCuando necesitás APIs, bases de datos o servicios de terceros
🔐 PermisosGuardarraíles de seguridadDefine qué puede y qué no puede hacer Claude en cada contexto
⚡ Slash CommandsAcciones manuales explícitasCuando vos decidís ejecutar algo, con control total del momento

🔗 Cómo se relacionan

Cada conversación con Claude Code
│
├── Carga automática al iniciar
│   ├── CLAUDE.md          → contexto permanente del proyecto
│   └── Skills relevantes  → conocimiento especializado por tarea
│
├── Se disparan solos (event-driven)
│   └── Hooks              → reaccionan a lo que Claude hace
│
├── Vos los activás manualmente
│   └── Slash Commands     → /deploy, /changelog, /standup
│
└── Disponibles como herramientas
    └── MCP Servers        → APIs, DBs, servicios externos

🎯 Regla de decisión rápida

NecesidadSolución
Claude no sabe nada del proyecto📄 Escribí un CLAUDE.md con el contexto
Claude no sabe cómo hacer algo🧩 Creá una Skill con las instrucciones
Querés que algo pase automáticamente🪝 Configurá un Hook para ese evento
Necesitás conectar con una API o DB🔌 Agregá un MCP Server
Querés ejecutar algo cuando vos querés⚡ Creá un Slash Command
Claude hace cosas que no debería🔐 Configurá Permisos en settings.json
✅
Estas piezas son complementarias, no alternativas. Un proyecto bien configurado usa todas: CLAUDE.md da el contexto, las skills el conocimiento, los hooks la automatización, MCP las conexiones externas y los permisos la seguridad.

📖 Glosario Términos clave

📌
Términos del ecosistema Claude Code — todo lo que necesitás entender para leer esta documentación sin googlear cada dos líneas.
Context Window
La cantidad máxima de texto (tokens) que Claude puede "ver" a la vez en una conversación. CLAUDE.md, skills activas, historial y archivos leídos compiten por este espacio limitado.
Si el contexto se llena, Claude puede "olvidar" instrucciones de más arriba en la conversación
Token
Unidad básica con la que los modelos de lenguaje procesan texto. Aproximadamente 3–4 caracteres en inglés. En español el tokenizer es menos eficiente — el mismo texto consume más tokens que en inglés. El context window se mide en tokens, no en palabras.
Un CLAUDE.md de 500 palabras consume ~650 tokens del contexto en cada conversación
Frontmatter
Bloque de metadatos al inicio de un archivo Markdown, delimitado por ---. En Claude Code se usa para definir el name, description y campos opcionales de skills y comandos.
El bloque ---name: pr-review--- al inicio de un SKILL.md es el frontmatter
Semantic Matching
El mecanismo por el cual Claude compara tu solicitud con las descripciones de las skills disponibles para decidir cuál activar. No busca palabras exactas — interpreta intención y significado.
"revisá el PR" puede activar una skill descrita como "reviews code changes before merge"
Skill
Archivo SKILL.md que le enseña a Claude cómo realizar una tarea específica. Se activa automáticamente cuando Claude detecta que tu solicitud coincide con su descripción. No requiere invocación manual.
Una skill de pr-review se activa sola cuando pedís revisar un PR
Agente
Instancia de Claude que opera con autonomía dentro de un contexto de ejecución: puede invocar herramientas, leer y escribir archivos, ejecutar comandos y encadenar acciones para completar una tarea compleja sin intervención continua.
Claude refactorizando un módulo entero por su cuenta, archivo por archivo, verificando que compile
Subagente
Instancia de Claude lanzada por el agente principal para ejecutar una tarea delegada en un contexto completamente aislado. Recibe instrucciones, trabaja de forma independiente y devuelve resultados sin "ver" la conversación principal.
El agente principal delega "escribir los tests" a un subagente mientras continúa con otra tarea
Execution Context
El entorno aislado en el que opera un agente o subagente: qué herramientas tiene disponibles, qué instrucciones recibió y cuál es el estado de su conversación. Cada subagente tiene el suyo propio, separado del principal.
El subagente de tests no "ve" el historial de conversación del agente principal
MCP (Model Context Protocol)
Protocolo abierto estándar que define cómo los modelos de lenguaje se comunican con herramientas y servicios externos. Permite a Claude conectarse a bases de datos, APIs y servicios sin código de integración personalizado.
Un MCP server de PostgreSQL permite a Claude ejecutar SQL queries directamente contra una base de datos
MCP Server
Proceso externo que implementa el protocolo MCP y expone capacidades (Tools, Resources, Prompts) que Claude puede usar durante una tarea. Se configura en settings.json y corre como proceso separado al iniciar Claude Code.
@modelcontextprotocol/server-github expone la API de GitHub como tools que Claude puede invocar
Hook
Script o comando que Claude Code ejecuta automáticamente cuando ocurre un evento específico — sin que el usuario lo pida. Los hooks son event-driven: se configuran una vez y se disparan solos cada vez que se cumple la condición.
Un hook PostToolUse corre el linter cada vez que Claude guarda un archivo
Event-driven
Paradigma donde una acción se dispara automáticamente en respuesta a un evento, sin intervención manual. Los hooks de Claude Code siguen este paradigma: reaccionan a acciones de Claude, no a pedidos del usuario.
El evento "Claude usó la herramienta Write" dispara automáticamente el hook de validación
Matcher
Expresión regular (regex) en la configuración de hooks que define contra qué herramienta o evento se compara. El carácter | es el operador OR de regex. Omitir el campo o usar "*" coincide con todo.
"matcher": "Write|Edit" activa el hook cuando Claude escribe o edita archivos (regex OR), pero no en Bash o Read
Slash Command
Instrucción predefinida que el usuario invoca manualmente escribiendo /nombre en la conversación. A diferencia de las skills, no se activan automáticamente — el usuario decide cuándo y con qué argumentos ejecutarlos.
/changelog 2.4.0 genera el changelog de esa versión pasando 2.4.0 como argumento
Progressive Disclosure
Técnica de organización donde solo se carga el contenido esencial inicialmente, y el detalle adicional se incorpora bajo demanda. Aplicado a skills: SKILL.md tiene lo esencial, la carpeta references/ tiene el detalle extenso.
La guía de arquitectura de 200 líneas solo se carga al contexto cuando alguien pregunta sobre arquitectura
allowed-tools
Campo opcional en el frontmatter de una skill que restringe qué herramientas nativas puede usar Claude mientras esa skill está activa. Útil para crear skills de solo lectura o con guardarraíles de seguridad.
allowed-tools: Read, Grep impide que Claude escriba archivos mientras la skill está activa
/doctor
Slash command de diagnóstico de Claude Code que verifica el estado del entorno de instalación: versión instalada, permisos del sistema, conectividad con la API y configuración activa. Primer paso ante cualquier problema técnico. Se ejecuta desde dentro de una sesión.
Si Claude Code falla silenciosamente, abrí una sesión y ejecutá /doctor para ver el diagnóstico
Settings Cascade
El sistema de resolución de configuración de Claude Code, donde múltiples archivos settings.json se combinan según su nivel (Enterprise → Personal → Project → Plugin), y las reglas más específicas prevalecen sobre las generales.
Una regla en .claude/settings.json del proyecto puede sobreescribir la configuración personal del desarrollador
CQRS
Command Query Responsibility Segregation. Patrón arquitectónico que separa las operaciones de escritura (Commands) de las de lectura (Queries) en clases o flujos distintos. Evita que los handlers mezclen responsabilidades.
CreateOrderCommandHandler escribe · GetOrderQueryHandler lee — nunca los dos en el mismo handler
Result<T>
Patrón de diseño (común en .NET, Rust, F#) que representa el resultado de una operación que puede fallar, sin usar excepciones para control de flujo. Retorna éxito con valor o error con detalle explícito.
Result<Order> puede ser Ok(order) o Fail("Cliente no encontrado") — sin try/catch

🪙 Tokens y Contexto Fundamental

💡
Idea clave: Entender tokens y contexto es entender los límites reales de Claude. Todo lo demás — skills, CLAUDE.md, subagentes — existe en parte para gestionar este recurso finito.

🔢 ¿Qué es un token?

Un token es la unidad mínima que el modelo procesa. No es una palabra ni un carácter — es un fragmento que puede ser parte de una palabra, una palabra entera o un signo de puntuación. El tokenizer divide el texto antes de enviarlo al modelo.

TextoTokens aprox.Nota
Una línea de código simple~10–20Varía mucho por lenguaje
Un párrafo de texto~100–150—
Un CLAUDE.md de 500 palabras~700Se consume en cada conversación
Un archivo de código de 300 líneas~2.500–4.000Depende de la densidad
Esta documentación completa~15.000—
📌
El tokenizer de Claude está optimizado para inglés. El mismo texto en español o en otros idiomas generalmente consume más tokens que en inglés — es decir, es menos eficiente.

📦 El context window

El context window es el "espacio de trabajo" del modelo — todo lo que Claude puede "ver" a la vez: el historial de la conversación, archivos leídos, CLAUDE.md, skills activas y resultados de herramientas. Es finito. Todo comparte el mismo espacio.

📌
Todos los modelos actuales de Claude tienen un context window de 200.000 tokens. Es grande, pero se llena más rápido de lo que parece en sesiones de trabajo intenso.

🫧 ¿Qué pasa cuando el contexto se llena?

Claude no te avisa con un error explícito — lo que pasa es más sutil y difícil de detectar:

1

Degradación gradual

Claude empieza a perder el foco en instrucciones que estaban al principio de la conversación. CLAUDE.md puede quedar fuera del "foco efectivo" aunque técnicamente siga cargado. Las respuestas se vuelven más genéricas y menos precisas.

2

Pérdida de coherencia

Claude empieza a contradecirse: aplica convenciones distintas a las establecidas, repite preguntas que ya respondiste, o ignora restricciones de CLAUDE.md que definiste al inicio de la sesión.

3

Corte forzado

En casos extremos, la API rechaza la request porque supera el límite máximo. Aparece un error explícito. Es el caso menos frecuente.

⚠️
El "olvido" de Claude en sesiones largas casi nunca es un bug del modelo — es el contexto cargado. Antes de asumir que Claude "no entiende", usá /compact o iniciá una sesión nueva con claude -c.

🗜️ /compact — la herramienta clave

El comando /compact comprime el historial de la conversación: el modelo genera un resumen denso de lo que se habló y reemplaza el historial completo por ese resumen. El resultado: libera espacio de contexto sin perder el hilo de la tarea.

/compactSesión nueva
Libera contexto✅ Sí✅ Sí
Mantiene el hilo de la tarea✅ Sí (resumido)❌ No
Mantiene CLAUDE.md y skills✅ Sí✅ Sí
Cuándo usarloSesión larga con contexto acumuladoTarea completamente nueva
 
/compact

# También podés pasar instrucciones de foco:
/compact focalizá en los cambios del módulo Orders
💡
No esperés a que el contexto explote. Comprimí proactivamente cuando sientas que la sesión lleva rato y Claude empieza a perder foco — no como rescate de último momento.

🏆 Estrategias para gestionar el contexto

EstrategiaCómoImpacto
📄 CLAUDE.md concisoMenos de 200 líneas, sin rellenoLibera tokens en cada conversación
🧩 Progressive disclosureMaterial extenso en references/, SKILL.md cortoSolo carga contexto cuando es necesario
🗜️ /compact proactivoComprimí antes de que el contexto se lleneMantiene coherencia en sesiones largas
🤖 SubagentesDelegá tareas independientes a contextos separadosEl principal no acumula todo el trabajo
✂️ Sesiones cortasUna sesión = un objetivo concretoEvita acumulación innecesaria
📁 No leer de másDejá que Claude pida los archivos cuando los necesiteCada Read consume tokens del contexto
✅
Regla de oro: El contexto es un recurso compartido entre vos, Claude y el proyecto. Gestionar bien CLAUDE.md, las skills y la duración de las sesiones es tan importante como escribir buen código.

🤖 Modelos Referencia

💡
Idea clave: Elegir el modelo correcto impacta velocidad, calidad y costo. La regla general: Opus para lo complejo y crítico, Sonnet para el desarrollo diario, Haiku para tareas simples y rápidas.

📊 Comparación de modelos

ModeloInteligenciaVelocidadCostoContext window
🟣 Claude Opus 4.6⭐⭐⭐⭐⭐ Máxima🐢 Más lento💰💰💰 Mayor200K tokens
🔵 Claude Sonnet 4.6⭐⭐⭐⭐ Muy alta⚡ Rápido💰💰 Medio200K tokens
🟢 Claude Haiku 4.5⭐⭐⭐ Near-frontier🚀 Más rápido💰 Menor200K tokens

🎯 ¿Cuándo usar cada uno?

TareaModelo recomendadoPor qué
Refactoring complejo, arquitectura, decisiones de diseño🟣 Opus 4.6Máxima capacidad para decisiones críticas
Desarrollo diario: coding, PR review, debugging🔵 Sonnet 4.6Balance ideal calidad / velocidad / costo
Análisis de código crítico (seguridad, migración)🟣 Opus 4.6No arriesgues calidad en tareas de alto impacto

Tareas repetitivas: formateo, renames, boilerplate🟢 Haiku 4.5No hace falta potencia máxima para esto
Preguntas rápidas, lookups, snippets simples🟢 Haiku 4.5Velocidad y costo mínimo

⚙️ Cómo cambiar el modelo

El modelo por defecto en Claude Code es Sonnet. Podés cambiarlo de varias formas:

MétodoComandoAlcance
Durante la sesión/model claude-opus-4-6Solo esa sesión
Al iniciarclaude --model claude-opus-4-6Solo esa sesión
En settings.json"model": "claude-sonnet-4-6"Permanente (User o Project scope)
En frontmatter de skillmodel: claude-haiku-4-5-20251001Solo cuando esa skill está activa
📌
IDs exactos de modelo: claude-opus-4-6 · claude-sonnet-4-6 · claude-haiku-4-5-20251001. Usá el ID completo en configuración para evitar ambigüedad.

💰 Tokens y costo en la práctica

En Claude Code, el costo se mide en tokens consumidos — tanto de entrada (historial, archivos, skills, CLAUDE.md) como de salida (lo que Claude genera). Cuanto más compleja la tarea y más contexto acumulado, más tokens consume cada turno.

EscenarioModelo idealPor qué
Sesión interactiva de desarrolloSonnetMuchos turnos → costo acumulado importa
Una tarea crítica puntualOpusPocos turnos → el costo extra se justifica
Agente corriendo en background sin intervenciónSonnetMuchos turnos autónomos → Opus sería muy caro
⚠️
No uses Opus para todo. Una sesión de refactoring masivo con Opus puede costar 10–20× más que con Sonnet para un resultado similar en tareas estándar. Reservá Opus para cuando realmente importa la máxima inteligencia.

🧩 Skills Referencia

💡
Idea clave: Si te encontrás explicándole lo mismo a Claude repetidamente, eso es una skill esperando ser escrita.

Una skill es un archivo markdown que le enseña a Claude cómo hacer algo una vez. Claude luego aplica ese conocimiento automáticamente cuando es relevante.

📌 ¿Qué son las Skills?

Las skills son carpetas de instrucciones que Claude Code puede descubrir y usar. Cada skill vive en un archivo SKILL.md con un nombre y descripción en su frontmatter. La descripción es cómo Claude decide si activarla.

---
name: pr-review
description: Reviews pull requests for code quality.
           Use when reviewing PRs or checking code changes.
---
# Instrucciones de revisión
(aquí van tus checklists, preferencias, etc.)

📁 ¿Dónde viven las Skills?

TipoUbicaciónAlcance
✅ Skills personales~/.claude/skillsTe siguen en todos tus proyectos
🏢 Skills de proyecto.claude/skills (raíz del repo)Compartidas con todo el equipo al clonar
🪟 Windows (personal)C:/Users/<usuario>/.claude/skillsSkills personales en Windows
📌
Las skills de proyecto se committean al control de versiones junto con tu código.

📂 Divulgación progresiva

Mantené las instrucciones esenciales en SKILL.md y el material de referencia en archivos separados que Claude lee solo cuando lo necesita. Regla de oro: SKILL.md bajo 500 líneas.

🤖 Agente Referencia

💡
Idea clave: Un agente no es solo Claude respondiendo preguntas — es Claude ejecutando tareas de forma autónoma, tomando decisiones y usando herramientas para llegar a un resultado concreto sin intervención continua del usuario.

📌 ¿Qué es un Agente?

En Claude Code, un agente es una sesión de Claude que puede invocar herramientas, leer y escribir archivos, ejecutar comandos y encadenar múltiples acciones para completar una tarea compleja. La diferencia con un chat normal es la autonomía: el agente decide qué hacer en cada paso, no espera instrucción por instrucción.

🔄 Ciclo de funcionamiento

  • 🎯 Recibe una tarea — del usuario o delegada por otro agente.
  • 🗺️ Planifica — descompone la tarea en pasos ejecutables.
  • ⚙️ Ejecuta herramientas — lee archivos, corre comandos, llama APIs.
  • 🔍 Evalúa resultados — analiza el output de cada acción.
  • 🔁 Itera — ajusta el plan según lo que encuentra, hasta terminar.

⚖️ Skill vs. Agente — la diferencia clave

📌
La confusión más común: una skill le enseña a Claude cómo hacer algo. Un agente lo hace. La skill da el conocimiento; el agente lo ejecuta con autonomía. Son complementarios, no alternativos.
🧩 Skill🤖 Agente
¿Qué es?Instrucciones en SKILL.md que Claude lee antes de actuarClaude actuando: planifica, decide, ejecuta, itera
¿Cómo se activa?Automáticamente por contexto semánticoCualquier sesión de Claude Code ya es un agente
¿Modifica archivos?No directamente — aporta el procedimientoSí, usa Read / Write / Edit / Bash autónomamente
¿Toma decisiones?No — es pasiva, solo guíaSí — elige qué herramienta usar y cuándo
¿Contexto propio?No, vive en el contexto del agente que la cargóSí, cada agente tiene su propio contexto
Caso de uso"Seguí este checklist al revisar PRs""Refactorizá todo el módulo Orders"
✅
La combinación ideal: Skill + Agente. La skill le da el contexto específico ("cómo aplicar CQRS en este proyecto") y el agente lo ejecuta de forma autónoma ("refactorizá OrderService.cs"). Sin skill, el agente improvisa. Sin agente, la skill no hace nada sola.

🌳 Árbol de agentes — orquestación

Cuando una tarea es demasiado grande para un solo contexto, o tiene partes independientes que pueden correr en paralelo, el agente principal puede delegar trabajo a subagentes. Esto forma un árbol de ejecución:

Agente Principal  →  "Migrá el módulo Orders a CQRS"
│
├── Subagente A  →  "Refactorizá OrderService.cs"
│                    contexto aislado · trabaja solo · devuelve resultado
│
├── Subagente B  →  "Escribí los tests para CreateOrderCommandHandler"
│                    contexto aislado · trabaja solo · devuelve resultado
│
└── Subagente C  →  "Revisá las migraciones de EF Core afectadas"
                     contexto aislado · trabaja solo · devuelve resultado

El agente principal no ejecuta los cambios directamente — delega, recibe los resultados de cada subagente y los integra o valida. Cada subagente trabaja sin saber que los otros existen.

⚠️
Los subagentes no comparten contexto. No ven la conversación del agente principal ni entre ellos. Solo reciben lo que el agente principal les pasa explícitamente en la instrucción de delegación.

🤖 Subagentes — contexto aislado

Un subagente es un agente lanzado por el principal con una tarea específica delegada. Su principal ventaja es el aislamiento: errores, historial o estado de un subagente no afectan al principal ni a otros subagentes.

CaracterísticaDetalle
Contexto propioNo ve el historial del agente principal ni de otros subagentes
Recibe solo lo necesarioEl principal le pasa exactamente las instrucciones y archivos relevantes para su tarea
Opera en paraleloMúltiples subagentes pueden correr al mismo tiempo sobre partes distintas del problema
Devuelve resultadosSu output (archivos modificados, reportes, etc.) vuelve al principal al terminar
Sin contaminaciónUn error o contexto cargado en un subagente no afecta el contexto del agente principal
💡
Cuándo usar subagentes: Cuando la tarea tiene partes independientes entre sí, cuando el contexto principal se está llenando con trabajo que podría aislarse, o cuando querés que distintas partes de la tarea usen skills diferentes sin interferencia.

🗂️ Tipos de Agentes

TipoDescripciónCaso de uso típico
👤 Agente principalLa sesión de Claude con la que interactuás directamenteDesarrollo interactivo, coordinación de tareas
🤖 SubagenteAgente lanzado por el principal para trabajo delegado en contexto aisladoTareas paralelas e independientes
🔁 Agente en loopAgente que itera hasta cumplir una condición de salida definidaCorrección de errores, refactoring progresivo

🛠️ Herramientas disponibles

HerramientaQué hace
ReadLee el contenido de un archivo
WriteEscribe o sobreescribe un archivo
EditModifica partes específicas de un archivo
BashEjecuta comandos de shell

⚠️
Un agente con acceso irrestricto a Write y Bash puede modificar cualquier archivo del proyecto. Definí siempre el alcance mínimo necesario usando allowed-tools en las skills o permisos en settings.json.

📄 CLAUDE.md Referencia

💡
Idea clave: CLAUDE.md es la memoria persistente de tu proyecto. Todo lo que escribís ahí, Claude lo sabe desde el primer mensaje de cada conversación.

📁 Dónde colocar CLAUDE.md

UbicaciónAlcanceCuándo carga
./CLAUDE.md o ./.claude/CLAUDE.mdTodo el proyectoSiempre que Claude corra en ese directorio
~/.claude/CLAUDE.mdTodos tus proyectos (global)Siempre, en cualquier proyecto
./src/CLAUDE.mdSolo ese subdirectorioOn-demand: solo cuando Claude trabaja dentro de src/
../CLAUDE.md (directorio padre)Proyectos en subdirectoriosSi Claude corre desde un subdirectorio, también carga los padres
📌
Carga aditiva por jerarquía: Claude combina todos los CLAUDE.md relevantes en cascada — global → proyecto → subdirectorio. Los más específicos tienen precedencia sobre los más generales. Los de subdirectorios cargan on-demand cuando Claude trabaja dentro de esa ruta.

✍️ ¿Qué poner en CLAUDE.md?

  • 🏗️ Arquitectura del proyecto
  • 📐 Estándares de código — estilo, naming conventions, linters

🪝 Hooks Referencia

💡
Idea clave: Los hooks son automatizaciones event-driven — se disparan solos cuando Claude hace algo, sin que vos lo pidás.

🔔 Eventos comunes

EventoCuándo se disparaCaso de uso típico
PreToolUseAntes de que Claude use una herramientaValidar parámetros, bloquear comandos peligrosos
PostToolUseDespués de que Claude usa una herramientaCorrer linter o build tras escribir/editar un archivo
NotificationCuando Claude envía una notificación al usuarioLogging, alertas externas, integración con Slack
StopCuando Claude termina de responderLimpieza, reportes post-tarea, notificaciones de fin

⚡ Eventos avanzados

📌
Estos eventos permiten un control más fino del ciclo de vida de la sesión y de los subagentes. Disponibles según versión y plataforma.
EventoCuándo se disparaCaso de uso típico
SessionStartAl iniciar una nueva sesión de Claude CodeCargar contexto adicional, logging de inicio
UserPromptSubmitCuando el usuario envía un mensajeEnriquecer el prompt automáticamente, auditoría
PermissionRequestCuando Claude pide permiso para una acciónAuto-aprobar ciertas acciones, logging de seguridad
PostToolUseFailureCuando una herramienta fallaManejo de errores, reintentos, alertas
SubagentStartAl lanzar un subagenteLogging de orquestación, inyectar contexto al subagente
SubagentStopCuando un subagente termina su tareaRecolectar resultados, integración post-delegación

🏗️ Estructura en settings.json

📌
El campo matcher es una expresión regular. El operador | funciona como OR de regex. Omitir matcher o usar "*" hace que el hook dispare ante cualquier herramienta. El campo opcional if permite un filtro adicional con sintaxis de permisos.
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{ "type": "command", "command": "npm run lint" }]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "if": "Bash(rm *)",
        "hooks": [{ "type": "command", "command": "/path/to/validate.sh" }]
      }
    ]
  }
}
⚠️
Los hooks corren comandos reales en el sistema. Probálos en entornos seguros antes de usarlos en producción — especialmente los de PreToolUse que pueden bloquear operaciones.

🔐 Permisos Seguridad

💡
Idea clave: Los permisos son los guardarraíles de Claude Code. Definen qué puede hacer Claude de forma autónoma, qué requiere tu confirmación y qué está directamente bloqueado.

🔑 Las tres reglas posibles

ReglaComportamientoCuándo usarla
allowClaude ejecuta sin pedir confirmaciónOperaciones seguras y repetitivas: leer archivos, correr tests, git status
askClaude muestra la acción y espera tu OKOperaciones que querés revisar antes: escribir archivos, commits, deploys
denyClaude no puede ejecutar esa acción, nuncaOperaciones peligrosas: rm -rf, comandos de producción, acceso a secretos

⚙️ Configuración en settings.json

{
  "permissions": {
    "allow": [
      "Read",
      "Bash(git *)",
      "Bash(dotnet test *)",
      "Bash(ng test *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)"
    ]
  }
}
📌
Los patrones en allow y deny soportan wildcards. Bash(git *) permite cualquier comando git. Bash(rm -rf *) bloquea específicamente ese patrón destructivo.

🖥️ Modos de sesión

Además de las reglas por herramienta, Claude Code tiene modos que controlan el nivel de autonomía en toda la sesión:

ModoComportamientoCaso de uso
defaultPide confirmación para acciones sensiblesDesarrollo diario normal
acceptEditsAcepta automáticamente ediciones de archivos sin confirmar una por unaRefactoring masivo donde confiás en el resultado
planClaude solo planifica y describe — no ejecuta ninguna acciónRevisar qué haría Claude antes de dejar que lo haga
autoMáxima autonomía — ejecuta todo sin confirmacionesPipelines CI/CD, entornos automatizados
⚠️
El modo auto equivale a --dangerously-skip-permissions. Solo usarlo en entornos automatizados controlados, nunca en desarrollo interactivo con acceso a producción.

🛠️ Gestionar permisos desde la sesión

/permissions

Muestra los permisos activos y permite agregar reglas allow o deny para la sesión actual sin tocar el settings.json.

🗂️ Scopes de Configuración Referencia

💡
Idea clave: Claude Code tiene cuatro niveles de configuración. Entender cuál aplica en cada contexto define qué se commitea al repo, qué es personal, y qué administra la organización.

📐 Los cuatro scopes

ScopeArchivo¿Se commitea?Quién lo controla
🏢 Managed Configuración central de la organización — IT / Administradores. Prevalece sobre todo. No editable por el usuario.
👤 User ~/.claude/settings.json ❌ No — es personal Vos. Aplica a todos tus proyectos. Preferencias personales, API keys, hooks globales.
📁 Project .claude/settings.json ✅ Sí — va al repo El equipo. Hooks, MCP, permisos y skills compartidos. Todo el equipo lo hereda al clonar.
🔒 Local .claude/settings.local.json ❌ No — Claude Code lo agrega a .gitignore automáticamente Vos, por proyecto. Tiene mayor prioridad que Project y User. Overrides locales: connection strings, tokens de desarrollo, prefs personales por repo.
⚠️
Nunca pongas secrets (tokens, passwords, connection strings reales) en .claude/settings.json — ese archivo va al repositorio. Usá siempre .claude/settings.local.json para valores sensibles.

🔁 Orden de precedencia

Cuando hay conflicto entre scopes, el orden de prevalencia (de mayor a menor) es:

Managed  >  Local  >  Project  >  User

Local sobreescribe Project y User — es el override personal por repositorio. Una regla deny en Managed no puede ser anulada por ningún otro scope.

📋 Qué va en cada scope — guía práctica

¿Qué configurar?Scope recomendado
Hooks de linting/build del proyecto📁 Project (.claude/settings.json)
MCP servers del proyecto (DB staging, GitHub)📁 Project (.claude/settings.json)
Connection string real de la base de datos local🔒 Local (.claude/settings.local.json)
Token personal de GitHub o API keys👤 User (~/.claude/settings.json)
Preferencias de modelo o formato de respuesta👤 User (~/.claude/settings.json)
Restricciones de seguridad para toda la organización🏢 Managed (administrado centralmente)

# EcommerceApi

## Stack
- .NET 8 · ASP.NET Core Minimal APIs
- Entity Framework Core 8 · SQL Server 2022
- xUnit · Moq · FluentAssertions

## Convenciones
- Usar record types para DTOs y Value Objects
- Métodos async terminan en Async
- Nunca retornar null: usar Result<T>
- EF Core: AsNoTracking() en queries de solo lectura

## Restricciones
- Nunca modificar migraciones existentes
- No usar DbContext en Controllers — solo en Repositories
- No hardcodear connection strings

## Comandos
- `dotnet build`
- `dotnet test --filter Category=Unit`
- `dotnet ef migrations add <Nombre>`
- `dotnet format`

🔧 Troubleshooting Errores comunes

📌
Los problemas más frecuentes al trabajar con Claude Code, organizados por síntoma. Hacé clic en cada uno para ver la causa y solución.

🧩 Skills

Mi skill no se activa cuando la pido Skill ▶
🔴
Síntoma: Claude no activa la skill aunque el pedido parece relevante.

Causas y soluciones:

  • 🔄 No reiniciaste Claude Code después de crear o editar la skill. Las skills cargan al inicio — siempre reiniciá la sesión tras cambios.
  • 📝 La descripción es muy vaga. Si dice solo "ayuda con código", Claude no sabe cuándo usarla. Agregá ejemplos concretos de frases que la activan: "Use when reviewing PRs, checking code before merge, or when asked for feedback on a branch."
  • 📁 El archivo está mal ubicado. La estructura debe ser .claude/skills/nombre-skill/SKILL.md — no .claude/skills/SKILL.md directamente.
  • ✏️ El frontmatter tiene errores. Los campos name y description son obligatorios y deben estar entre los dashes ---.

Verificación rápida:

/skills

Si tu skill no aparece en la lista, el problema es de ubicación o frontmatter.

Se activa la skill incorrecta cuando tengo dos con nombres similares Skill ▶
🔴
Síntoma: Claude activa la skill del proyecto en vez de la personal, o viceversa.

El orden de prioridad es: Enterprise → Personal → Project → Plugins. Si tenés dos skills con el mismo name, gana la de mayor prioridad.

  • Usá nombres descriptivos para evitar colisiones: frontend-review en vez de review.
  • Si querés que la de proyecto sobreescriba la personal, la única opción es renombrar la personal.
  • Usá /skills para ver el origen (personal/proyecto) de cada skill activa.
La skill carga pero Claude no sigue las instrucciones correctamente Skill ▶
🔴
Síntoma: Claude confirma que activó la skill pero el resultado no respeta el formato o los pasos definidos.
  • 📏 SKILL.md es demasiado largo. Si supera las 500 líneas, Claude puede omitir partes. Mové el material extenso a archivos en references/ y referencialos desde el SKILL.md.
  • ⚔️ Conflicto con CLAUDE.md. Si CLAUDE.md tiene instrucciones que contradicen la skill, CLAUDE.md puede ganar por estar siempre en contexto. Revisá que no haya solapamiento.
  • 🔢 Instrucciones ambiguas. Los pasos numerados y las listas con viñetas funcionan mejor que los párrafos largos para instrucciones procedurales.

🪝 Hooks

El hook no se ejecuta después de que Claude edita un archivo Hook ▶
🔴
Síntoma: Claude escribe o edita archivos pero el hook no dispara.
  • 🔄 Reiniciá Claude Code. Los hooks se leen al iniciar la sesión.
  • 📄 Verificá el JSON. Un error de sintaxis en settings.json silencia todos los hooks. Validá con claude config list.
  • 🎯 Revisá el matcher. "matcher": "Write" solo activa en writes. Para edits también, usá "matcher": "Write|Edit".
  • 📁 Ubicación del settings.json. Debe estar en .claude/settings.json (proyecto) o ~/.claude/settings.json (global).
// Verificar configuración activa
claude config list
El hook falla con "command not found" o errores de path Hook ▶
🔴
Síntoma: El hook dispara pero el comando no se encuentra o falla con error de ruta.
  • 🛣️ Los hooks corren con el PATH del sistema, no del shell interactivo. Usá rutas absolutas o verificá que el comando esté en el PATH del sistema.
  • 📦 El comando requiere contexto del proyecto. Usá cd /ruta/al/proyecto && npm run lint si el comando necesita estar en un directorio específico.
  • 🔑 Permisos de ejecución. Si el hook llama a un script, verificá que tenga permisos: chmod +x script.sh.

🔌 MCP

El MCP server aparece como desconectado MCP ▶
🔴
Síntoma: /mcp muestra el server como offline o Claude dice que no puede conectarse.
  • 📦 El paquete no está instalado. Con npx -y se instala automáticamente, pero puede fallar en redes corporativas. Intentá instalarlo manualmente: npm install -g @modelcontextprotocol/server-postgres.
  • 🔑 Variables de entorno faltantes. Verificá que las variables en el bloque env estén definidas y sean correctas.
  • 🔒 Problema de conectividad. Si la DB o API es remota, verificá que el host sea accesible desde tu máquina con un ping o telnet.
  • 📋 Revisá los logs del server:
claude --verbose

En modo verbose verás los errores de inicialización del MCP server.

📄 CLAUDE.md / General

Claude ignora las instrucciones de CLAUDE.md CLAUDE.md ▶
🔴
Síntoma: Claude no respeta convenciones o restricciones definidas en CLAUDE.md.
  • 📏 CLAUDE.md es demasiado largo. Si tiene cientos de líneas, las instrucciones al final pueden quedar fuera del foco. Priorizar y condensar.
  • ⚔️ Instrucción contradictoria en el chat. Si el usuario da una instrucción contraria a CLAUDE.md en el mismo mensaje, puede prevalecer. CLAUDE.md establece defaults, no reglas absolutas.
  • 📁 Archivo mal nombrado. Debe ser exactamente CLAUDE.md (mayúsculas) en la raíz del proyecto o en el directorio home.
El contexto se llena rápido y Claude empieza a "olvidar" cosas General ▶
🔴
Síntoma: En sesiones largas, Claude empieza a ignorar instrucciones o perder el hilo de la tarea.
  • 🗜️ Usá /compact para comprimir el historial de la conversación sin perder el contexto esencial.
  • 📂 Mové contenido de CLAUDE.md a skills. Lo que no necesita estar siempre presente consume contexto innecesariamente.
  • ✂️ Dividí tareas largas. En vez de una sesión gigante, usá subagentes o sesiones separadas para módulos distintos.
  • 🔄 Iniciá una sesión nueva con claude --continue para retomar el contexto resumido en vez de acumularlo indefinidamente.
