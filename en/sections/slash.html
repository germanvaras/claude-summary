⚡ Slash Commands Manual control↗ Docs

💡
Key concept: If skills are the autopilot, slash commands are the manual joystick. You control them, whenever you want. There are three distinct categories with different origins.

1️⃣ Built-in commands — system commands

Commands that Claude Code ships with by default. They do not need configuration — they are always available within a session.

/helpLists all available commands in the current session
/statusShows configuration state: active model, directory, loaded skills and permissions
/clearClears the conversation history (aliases: /reset, /new)
/compact [instructions]Compresses history to free context space. You can pass focus instructions: /compact focus on the Orders module changes
/contextVisualize token usage of the current context
/configOpens the configuration interface (alias: /settings)
/permissionsView and update permissions for the current session (alias: /allowed-tools)
/model [model]View or change the model for this session
/initGenerates an initial CLAUDE.md by analyzing the current project
/memoryEdit CLAUDE.md files and project memory
/mcpView the status of connected MCP servers
/skillsLists all active skills for the current directory
/hooksView active hook configurations
/agentsView agent configurations
/doctorFull diagnostic: version, authentication, connectivity
/plan [description]Enter planning mode — Claude describes what it would do but does not execute anything
/resume [session]Resume a previous conversation (alias: /continue)
/exitClose the session (alias: /quit)
⚠️
Not all built-in commands are available in all contexts. Availability depends on the plan, platform (CLI, VS Code, etc.), and installed version. Use /help to see which ones are active in your session.

2️⃣ Skill commands — commands you create

Custom commands defined in .md files inside commands/ folders. A file at .claude/commands/deploy.md creates the /deploy command. The difference from a skill: the user invokes them manually, they do not activate by semantic context.

TypeLocationScope
Personal~/.claude/commands/name.mdAll your projects
Project.claude/commands/name.mdShared with the team on clone

Custom command structure:

# .claude/commands/daily-standup.md

Generate a standup summary based on:
1. Commits from the last 24 hours
2. Modified files
3. New TODOs or FIXMEs

## ✅ What I did yesterday
## 🎯 What I'm doing today
## 🚧 Blockers or questions

The user invokes it simply with:

/daily-standup

Passing arguments with $ARGUMENTS:

Use $ARGUMENTS to capture all the text that follows the command name:

# .claude/commands/review.md

Review the code on the $ARGUMENTS branch:
1. Check naming conventions
2. Review error handling and edge cases
3. Verify there are sufficient tests
4. List required changes or approve

Invocation:

/review feature/login
# $ARGUMENTS = "feature/login"

/review feature/order-refactor --strict
# $ARGUMENTS = "feature/order-refactor --strict"
📌
$ARGUMENTS captures everything that follows the command name as a string. If you need multiple parameters, you can use your own conventions within the command text (e.g. parse by space or dash) and describe the expected format in the command file.

3️⃣ MCP prompts — commands from external servers

MCP servers can expose predefined prompts that appear as slash commands in the format /mcp__<server>__<prompt>. They are provided by the server — you do not create them.

/mcp__github__create-pr
/mcp__postgres__explain-query
/mcp__slack__post-summary
📌
Available MCP prompts appear in /help when the corresponding MCP server is connected and active.
