🧩 Skills Specialized knowledge↗ Docs

💡
Key concept: If you find yourself explaining the same thing to Claude repeatedly, that is a skill waiting to be written. A skill is written once and applies itself whenever it is relevant.

📌 What are Skills?

A skill is a SKILL.md file inside its own folder that teaches Claude how to perform a specific task. Claude automatically detects when to activate it by comparing your request with the skill's description (semantic matching). You do not need to invoke it manually.

🚀 Create your first skill — step by step

1

Create the folder structure

Each skill lives in its own folder inside .claude/skills/:

# Windows PowerShell
mkdir .claude\skills\pr-review

# Linux / macOS / Git Bash
mkdir -p .claude/skills/pr-review
2

Create the SKILL.md file with frontmatter

# .claude/skills/pr-review/SKILL.md
---
name: pr-review
description: Reviews pull requests for code quality, security and conventions.
             Use when reviewing PRs, checking code before merge, giving feedback
             on a branch, or when asked to review code changes.
---

## Review process

1. Read the full diff before commenting
2. Check: naming conventions, error handling, tests
3. Indicate severity: [blocking] / [suggestion] / [nitpick]
4. If there are security issues, mark them as [critical]
5. At the end: summary with approval or list of required changes
📌
The description is the most important field. Claude uses it to decide when to activate the skill. Include verbs and phrases users will write: "reviewing PRs", "checking code before merge", "give feedback on". Max 1,024 characters.
3

Restart the Claude Code session

Skills are loaded at session start. After creating or modifying a skill, close the session and open a new one:

/exit
# then in the terminal:
claude
4

Verify the skill loaded

/skills

pr-review should appear in the list with its description and origin (personal or project).

If it doesn't appear: check the file location (it must be .claude/skills/pr-review/SKILL.md, not .claude/skills/SKILL.md directly) and that the frontmatter has the name and description fields.

5

Test the skill

"review the PR on the feature/login branch"

Claude should activate the pr-review skill automatically. If it doesn't, the description may be too vague — add more example phrases.

📁 Where do Skills live?

TypeLocationScope
Project skills.claude/skills/name/SKILL.md (at repo root)Shared with the entire team on clone. Committed to the repo.
Personal skills~/.claude/skills/name/SKILL.mdFollow you across all your projects. Not committed.
Windows (personal)C:\Users\<user>\.claude\skills\name\SKILL.mdEquivalent to personal on Windows.

MechanismWhen it loadsHow it activatesIdeal use case

🏆 Skill priority

When two skills have the same name, the one with higher priority wins:

PriorityTypeLocation
1 (highest)Managed (Enterprise)Organization-administered configuration
2Personal (User)~/.claude/skills/
3Project.claude/skills/ in the repo
4 (lowest)PluginsSkills from installed plugins

🔧 Frontmatter fields

FieldRequiredDescription
name✅ YesUnique identifier. Lowercase, numbers, and hyphens only. Max 64 characters.
description✅ YesTells Claude when to use the skill. The more specific and the more example phrases, the better. Max 1,024 characters.
allowed-tools❌ NoRestricts which tools Claude can use while the skill is active. Useful for read-only skills.
model❌ NoSpecifies which model to use when this skill is active. E.g.: claude-haiku-4-5-20251001 for simple task skills.

📂 Progressive disclosure with references/

If your skill needs extensive guides, don't put them all in SKILL.md — that always occupies context. Use the references/ folder:

.claude/skills/refactor-dotnet/
├── SKILL.md          ← essential instructions (<500 lines)
└── references/
    ├── patterns.md   ← extensive CQRS, Repository guides, etc.
    └── examples.md   ← reference code examples

In SKILL.md, reference them explicitly so Claude reads them when needed:

---
name: refactor-dotnet
description: Refactors C# / .NET 8 code...
---

## Base instructions
1. Read the full file before modifying
2. Apply patterns from references/patterns.md when applicable
3. When done: `dotnet build` and list changes

## Reference
- Detailed patterns: @references/patterns.md
- Examples: @references/examples.md
📌
Claude only loads the content of references/ when it needs it for the task. SKILL.md always loads; references/ loads on-demand. This avoids filling the context with material that is not always necessary.

🔄 Multiple skills active simultaneously

Claude can activate more than one skill at the same time if the task matches several. Instructions are combined in context. If there are conflicts between instructions from different skills, CLAUDE.md takes precedence, and among skills the one with higher priority in the table above wins.

⚠️
If you have skills with contradictory instructions (e.g. one says "use tabs" and another "use spaces"), the result may be inconsistent. Design skills to be complementary, not overlapping.
