🔐 Permissions Security↗ Docs

💡
Key concept: Permissions are Claude Code's guardrails. They define what Claude can do autonomously, what requires your confirmation, and what is outright blocked — both at the configuration file level and per session.

🔑 The three possible rules

RuleBehaviorWhen to use it
allowClaude executes without asking for confirmationSafe, repetitive operations: reading files, running tests, git status
askClaude shows the action and waits for your OK before executingOperations you want to review: writing new files, commits, deploys
denyClaude cannot execute that action under any circumstancesDangerous operations: rm -rf, production commands, access to secrets

⚙️ Configuration in settings.json

Complete example with all three rules:

{
  "permissions": {
    "allow": [
      "Read",
      "Bash(git status)",
      "Bash(git log *)",
      "Bash(git diff *)",
      "Bash(dotnet test *)",
      "Bash(npm run test *)"
    ],
    "ask": [
      "Write",
      "Edit",
      "Bash(git commit *)",
      "Bash(git push *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Bash(DROP *)",
      "Bash(kubectl delete *)"
    ]
  }
}
📌
Patterns support wildcards. Bash(git *) covers any git subcommand. Bash(dotnet test *) covers any dotnet test variant. The more specific the pattern, the more control you have.

🖥️ Session modes

In addition to per-tool rules, Claude Code has modes that control the autonomy level for the entire session. Modes overlay the settings.json rules — deny rules are always respected regardless of mode.

ModeCLI flagBehaviorUse case
default(no flag)Asks for confirmation on sensitive actions per settings.json rulesNormal daily development
plan--permission-mode planClaude only plans and describes — does not execute any actionReview what Claude would do before letting it proceed
auto--dangerously-skip-permissionsMaximum autonomy — executes everything without confirmations (except deny rules)CI/CD pipelines, automated environments
⚠️
--dangerously-skip-permissions bypasses all interactive confirmations. Only use it in controlled automated environments. deny rules in settings.json remain active — they are the only guardrail that works in auto mode.

🔄 Interaction between modes and settings.json rules

File rules and session mode operate on separate layers:

ScenarioResult
Action in allow + default modeExecutes without confirmation
Action in ask + default modeClaude asks before executing
Action in ask + auto modeExecutes without confirmation (auto mode overrides ask)
Action in deny + auto modeBlocked — deny always overrides the mode
Action with no rule + default modeClaude asks for confirmation (conservative default behavior)

🛠️ Managing permissions from the session

You can view and add rules in real time without touching settings.json:

/permissions

Shows active permissions and allows adding allow, ask, or deny rules for the current session. Changes made with /permissions are session-only — they do not persist to settings.json.

✅
Recommended strategy: Start with conservative rules (little in allow, much in ask) and loosen as you gain confidence in Claude's behavior in your project. deny rules for destructive operations should always be there.
