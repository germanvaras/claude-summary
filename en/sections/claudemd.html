📄 CLAUDE.md Project memory↗ Docs

💡
Key concept: CLAUDE.md is your project's persistent memory. Everything you write there, Claude knows from the very first message of every conversation — without you having to repeat it.

📁 Locations and differences

LocationDescriptionWhen it loadsScope
./CLAUDE.md At the project root, visible in the repository Whenever Claude runs in that directory Entire project
./.claude/CLAUDE.md Inside the .claude/ folder. Equivalent in scope to ./CLAUDE.md. Useful for keeping the root clean. Whenever Claude runs in that directory Entire project
~/.claude/CLAUDE.md Personal global CLAUDE.md. Applies to all your projects. For your personal work preferences. Always, in any project All your projects
./src/CLAUDE.md Instructions specific to a subdirectory. Only applies when Claude works inside src/. On-demand: only when Claude works inside that directory That subdirectory only
📌
CLAUDE.md vs .claude/CLAUDE.md? They are equivalent in scope — both load whenever Claude runs from that directory. The difference is visibility: CLAUDE.md at the root is more visible to the team; .claude/CLAUDE.md groups all Claude configuration in one place. Choose based on your project's convention.
📌
Additive cascade loading: Claude combines all relevant CLAUDE.md files in cascade — global → project → subdirectory. More specific ones take precedence if there are contradictions. Subdirectory ones load on-demand.

✍️ What to put in CLAUDE.md?

  • 🏗️ Project architecture — layer structure, patterns used, important decisions
  • 📐 Code standards — style, naming conventions, linters, formatters
  • 🚫 Explicit restrictions — "never modify migrations", "don't use DbContext in controllers"
  • 🛠️ Stack and common commands — build, test, run, deploy
  • 🔗 Business context — domain glossary, key entities

📏 Recommended length

CLAUDE.md loads in every conversation and consumes context tokens. Keep it under 200 lines. Extensive material belongs in skills with progressive disclosure:

ContentWhere to put it
Project rules and conventions (always needed)CLAUDE.md
Detailed architecture guides (only when refactoring)Skill → references/
PR review checklistsSkill pr-review
Internal API documentationSkill → references/

💡 Complete real-world example

# EcommerceApi

## Stack
- .NET 8 · ASP.NET Core Minimal APIs
- Entity Framework Core 8 · SQL Server 2022
- xUnit · Moq · FluentAssertions

## Architecture
Clean Architecture: Domain → Application → Infrastructure → API
- Commands and Queries separated (CQRS)
- Repositories abstract EF Core from Application

## Code conventions
- Record types for DTOs and Value Objects
- Async methods end in Async
- Never return null: use Result<T> from FluentResults
- EF Core: AsNoTracking() on read-only queries

## Restrictions
- Never modify existing migrations — create a new migration
- Don't use DbContext in Controllers or Application — only in Repositories
- Don't hardcode connection strings

## Commands
- build:   `dotnet build`
- test:    `dotnet test --filter Category=Unit`
- lint:    `dotnet format`
- migrate: `dotnet ef migrations add <Name>`

🚀 Auto-generate one with /init

If the project already has code, Claude can analyze the repository and generate an initial CLAUDE.md:

/init

Claude reads the project structure, configuration files (package.json, .csproj, pyproject.toml, etc.) and proposes a CLAUDE.md with the detected stack and conventions. Review and edit the result before committing it — Claude can detect the stack correctly but does not know your business restrictions.

⚠️
An instruction in the chat can override CLAUDE.md if it directly contradicts what is written there. CLAUDE.md establishes default behavior, not absolute rules. For restrictions that must be impossible to bypass, use deny rules in settings.json permissions.
