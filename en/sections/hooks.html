🪝 Hooks Automation↗ Docs

💡
Key concept: Hooks are event-driven automations — they trigger on their own when Claude does something, without you asking. Configure once, works always.

🔔 Available events

EventWhen it firesTypical use case
PreToolUseJust before Claude uses a toolValidate parameters, block dangerous commands, audit logging
PostToolUseAfter Claude uses a tool (success)Run linter or build after writing/editing, notifications
NotificationWhen Claude sends a notification to the userLogging, external alerts, Slack integration
StopWhen Claude finishes respondingCleanup, post-task reports, completion notifications
UserPromptSubmitWhen the user submits a messageAutomatically enrich the prompt, input auditing

🏗️ Structure in settings.json

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          { "type": "command", "command": "npm run lint" }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "/absolute/path/validate.sh" }
        ]
      }
    ]
  }
}
📌
The matcher field is a regular expression compared against the tool name. The | operator works as regex OR: "Write|Edit" activates the hook on Write or Edit. Omitting matcher makes the hook fire on any tool.

📥 What information the hook receives

Claude Code passes the event information to the hook process via stdin as JSON. The hook can read that JSON to know exactly what Claude did.

Structure for PreToolUse and PostToolUse:

# stdin received by the hook:
{
  "tool_name": "Write",
  "tool_input": {
    "file_path": "/src/OrderService.cs",
    "content": "..."
  }
}

# For PostToolUse, also includes the response:
{
  "tool_name": "Write",
  "tool_input": { "file_path": "..." },
  "tool_response": { "success": true }
}

Example hook script using this information:

#!/bin/bash
# validate-before-write.sh
INPUT=$(cat)   # reads the JSON from stdin
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path')

if [[ "$FILE" == *"migrations"* ]]; then
  echo "ERROR: Migration files cannot be modified directly."
  exit 2   # exit 2 = BLOCK the action
fi

exit 0   # exit 0 = ALLOW the action

🚦 How PreToolUse can block actions

A PreToolUse hook can decide whether Claude can execute the tool, based on its exit code:

Exit codeBehaviorWhen to use it
0Action allowed, Claude continues normallySuccessful validation
2Action blocked. What the hook wrote to stdout is shown to Claude as an explanation.When you detect something Claude should not execute
Other (≠0, ≠2)Hook error — Claude receives an error notification but may continueUnexpected script failure
✅
What the hook writes to stdout with exit 2 reaches Claude as feedback. Claude reads it and can adjust its plan. Write clear messages: "Migration file protected — use the create-migration.sh script".

❌ What happens when a hook fails

Hook typeIf it fails (exit code other than 0 and 2)
PreToolUseClaude receives a hook error notification. The action may continue or stop depending on configuration.
PostToolUseClaude receives the error output and can react — for example, if the linter fails, Claude can see the errors and attempt to fix them.
StopThe error is logged but does not interrupt the flow — the end hook cannot undo what Claude already did.

🛣️ Environment variables in hooks

Hooks run with the system PATH, not the interactive shell's PATH. This causes most "command not found" errors in hooks. Practical rules:

  • Use absolute paths for executables: /usr/local/bin/npm instead of npm
  • If the command needs to be in the project directory, prefix with cd /project/path &&
  • Hooks inherit the environment variables of the Claude Code process, including those from your terminal session
  • On Windows, use where npm to find the full path and use it in the hook
# ❌ May fail — npm is not in the hook's PATH
"command": "npm run lint"

# ✅ Absolute path — always works
"command": "C:\\Users\\user\\AppData\\Roaming\\npm\\npm.cmd run lint"

# ✅ Or change to the project directory first
"command": "cd C:\\project && npm run lint"

📋 Practical examples

Automatic linter after editing TypeScript files:

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "npx eslint --fix {tool_input.file_path}" }]
    }]
  }
}

Automatic build after .NET changes:

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "dotnet build --no-restore -q 2>&1 | tail -5" }]
    }]
  }
}

Notification when a long task finishes:

{
  "hooks": {
    "Stop": [{
      "hooks": [{ "type": "command", "command": "powershell -Command \"[System.Media.SystemSounds]::Beep.Play()\"" }]
    }]
  }
}
⚠️
Hooks run real commands on the system. Test them with claude --verbose in development environments before using them in production. A badly written PreToolUse hook can block all Claude operations.
