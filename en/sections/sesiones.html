🏷️ Sessions Workflow↗ Docs

💡
Key concept: One session = one task. Keeping sessions separate by objective keeps context clean, reduces cost per turn, and lets you resume work exactly where you left off.

🔄 Session commands

CommandWhat it doesWhen to use it
claudeStarts a new session with clean contextEvery time you start a new task
claude -c
claude --continue		Resumes the last active session from where it left offYou want to continue what you were doing
claude -r <session-id>
claude --resume <session-id>		Resumes a specific session by its IDYou want to return to a previous task, not the last one
claude -n <name>
claude --session-name <name>		Starts a new session with a custom nameYou want to easily identify the session without relying on the auto-generated ID

🏷️ How to identify and resume sessions

Each session has a unique auto-generated ID. To be able to resume a specific session later:

1

Describe the objective at the start

The first message of the session is the context anchor. If you write it clearly — what you're building, what constraints exist, what you've decided — when you resume with --resume, Claude picks up the thread without you having to re-explain everything.

> "I'm migrating the authentication module from JWT to OAuth2.
> Stack: Node.js + Express. User model is done.
> Remaining: token validation middleware and protected routes."
2

Find the session-id

The session ID appears in the Claude Code prompt at startup or when listing sessions. You can also use:

claude --list-sessions

This shows recent sessions with their ID, date, and first message.

3

Resume with the ID or name

claude --resume abc123def       # by auto-generated ID
claude --resume my-task         # by name assigned with -n

Claude loads the full history of that session and continues as if there had been no interruption.

🗜️ Long session: use /compact before resuming

If a session has many accumulated turns, it is worth compressing it before resuming with --resume. This way, the loaded context is dense and relevant, not the raw full history.

/compact focus on the current state of the auth module

✂️ Why one session per task

One session per taskEverything in one session
Context✅ Clean and relevant❌ Accumulated from previous tasks
Cost per turn✅ Lower (fewer input tokens)❌ Grows with each new task
Claude coherence✅ Focused on the current task❌ Risk of mixing context from different tasks
Ability to resume✅ Resume exactly that task❌ Hard to know where each thing was left
⚠️
Verify flags in your version. The --resume and --list-sessions commands are documented in the CLI. You can always run claude --help to see all available flags in your installed version.
