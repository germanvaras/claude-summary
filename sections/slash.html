⚡ Slash Commands Control manual

💡
Idea clave: Si las skills son el piloto automático, los slash commands son el joystick manual. Los controlás vos, cuando querés. Existen tres categorías distintas con origen diferente.

1️⃣ Built-in commands — los del sistema

Comandos que Claude Code trae por defecto. No se configuran — están siempre disponibles dentro de una sesión.

/helpLista todos los comandos disponibles en la sesión actual
/statusMuestra el estado de configuración: modelo activo, directorio, skills y permisos cargados
/clearLimpia el historial de la conversación (aliases: /reset, /new)
/compact [instrucciones]Comprime el historial para liberar espacio de contexto. Podés pasar instrucciones de foco: /compact focalizá en los cambios del módulo Orders
/contextVisualizar el uso de tokens del contexto actual
/configAbre la interfaz de configuración (alias: /settings)
/permissionsVer y actualizar permisos para la sesión actual (alias: /allowed-tools)
/model [modelo]Ver o cambiar el modelo para esta sesión
/initGenera un CLAUDE.md inicial analizando el proyecto actual
/memoryEditar archivos CLAUDE.md y memoria del proyecto
/mcpVer el estado de los servidores MCP conectados
/skillsLista todas las skills activas para el directorio actual
/hooksVer las configuraciones de hooks activas
/agentsVer configuraciones de agentes
/doctorDiagnóstico completo: versión, autenticación, conectividad
/plan [descripción]Entrar en modo planificación — Claude describe qué haría pero no ejecuta nada
/resume [sesión]Retomar una conversación anterior (alias: /continue)
/exitCierra la sesión (alias: /quit)
⚠️
No todos los built-in commands están disponibles en todos los contextos. La disponibilidad depende del plan, la plataforma (CLI, VS Code, etc.) y la versión instalada. Usá /help para ver los que tenés activos en tu sesión.

2️⃣ Skill commands — los que creás vos

Comandos custom definidos en archivos .md dentro de las carpetas commands/. Un archivo en .claude/commands/deploy.md crea el comando /deploy. La diferencia con una skill: el usuario los invoca manualmente, no se activan por contexto semántico.

TipoUbicaciónAlcance
Personales~/.claude/commands/nombre.mdTodos tus proyectos
De proyecto.claude/commands/nombre.mdCompartidos con el equipo al clonar

Estructura de un comando custom:

# .claude/commands/daily-standup.md

Generá un resumen de standup basado en:
1. Commits de las últimas 24 horas
2. Archivos modificados
3. TODOs o FIXMEs nuevos

## ✅ Qué hice ayer
## 🎯 Qué voy a hacer hoy
## 🚧 Bloqueos o dudas

El usuario lo invoca simplemente con:

/daily-standup

Pasar argumentos con $ARGUMENTS:

Usá $ARGUMENTS para capturar todo el texto que siga al nombre del comando:

# .claude/commands/review.md

Revisá el código de la rama $ARGUMENTS:
1. Verificá convenciones de naming
2. Revisá manejo de errores y edge cases
3. Verificá que haya tests suficientes
4. Listá los cambios requeridos o aprobá

Invocación:

/review feature/login
# $ARGUMENTS = "feature/login"

/review feature/order-refactor --strict
# $ARGUMENTS = "feature/order-refactor --strict"
📌
$ARGUMENTS captura todo lo que sigue al nombre del comando como un string. Si necesitás múltiples parámetros, podés usar convenciones propias dentro del texto del comando (ej: parsear por espacio o guion) y describir el formato esperado en el archivo del comando.

3️⃣ MCP prompts — comandos de servidores externos

Los servidores MCP pueden exponer prompts predefinidos que aparecen como slash commands con el formato /mcp__<servidor>__<prompt>. Los provee el servidor — no los creás vos.

/mcp__github__create-pr
/mcp__postgres__explain-query
/mcp__slack__post-summary
📌
Los MCP prompts disponibles aparecen en /help cuando el servidor MCP correspondiente está conectado y activo.
