🔐 Permisos Seguridad

💡
Idea clave: Los permisos son los guardarraíles de Claude Code. Definen qué puede hacer Claude de forma autónoma, qué requiere tu confirmación y qué está directamente bloqueado — tanto a nivel de archivo de configuración como de sesión.

🔑 Las tres reglas posibles

ReglaComportamientoCuándo usarla
allowClaude ejecuta sin pedir confirmaciónOperaciones seguras y repetitivas: leer archivos, correr tests, git status
askClaude muestra la acción y espera tu OK antes de ejecutarOperaciones que querés revisar: escribir archivos nuevos, commits, deploys
denyClaude no puede ejecutar esa acción bajo ninguna circunstanciaOperaciones peligrosas: rm -rf, comandos de producción, acceso a secretos

⚙️ Configuración en settings.json

Ejemplo completo con las tres reglas:

{
  "permissions": {
    "allow": [
      "Read",
      "Bash(git status)",
      "Bash(git log *)",
      "Bash(git diff *)",
      "Bash(dotnet test *)",
      "Bash(npm run test *)"
    ],
    "ask": [
      "Write",
      "Edit",
      "Bash(git commit *)",
      "Bash(git push *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Bash(DROP *)",
      "Bash(kubectl delete *)"
    ]
  }
}
📌
Los patrones soportan wildcards. Bash(git *) cubre cualquier subcomando de git. Bash(dotnet test *) cubre cualquier variante de dotnet test. Cuanto más específico el patrón, más control tenés.

🖥️ Modos de sesión

Además de las reglas por herramienta, Claude Code tiene modos que controlan el nivel de autonomía en toda la sesión. Los modos se solapan con las reglas de settings.json — las reglas deny siempre se respetan independientemente del modo.

ModoFlag CLIComportamientoCaso de uso
default(sin flag)Pide confirmación para acciones sensibles según las reglas de settings.jsonDesarrollo diario normal
plan--permission-mode planClaude solo planifica y describe — no ejecuta ninguna acciónRevisar qué haría Claude antes de dejar que lo haga
auto--dangerously-skip-permissionsMáxima autonomía — ejecuta todo sin confirmaciones (excepto deny rules)Pipelines CI/CD, entornos automatizados
⚠️
--dangerously-skip-permissions omite todas las confirmaciones interactivas. Solo usarlo en entornos automatizados controlados. Las reglas deny en settings.json siguen activas — son el único guardarraíl que funciona en modo auto.

🔄 Interacción entre modos y reglas de settings.json

Las reglas del archivo y el modo de sesión funcionan en capas distintas:

EscenarioResultado
Acción en allow + modo defaultSe ejecuta sin confirmación
Acción en ask + modo defaultClaude pregunta antes de ejecutar
Acción en ask + modo autoSe ejecuta sin confirmación (el modo auto sobreescribe ask)
Acción en deny + modo autoBloqueada — deny siempre prevalece sobre el modo
Acción sin regla + modo defaultClaude pide confirmación (comportamiento conservador por defecto)

🛠️ Gestionar permisos desde la sesión

Podés ver y agregar reglas en tiempo real sin tocar settings.json:

/permissions

Muestra los permisos activos y permite agregar reglas allow, ask o deny para la sesión actual. Los cambios hechos con /permissions son solo para esa sesión — no persisten al settings.json.

✅
Estrategia recomendada: Empezá con reglas conservadoras (poco en allow, mucho en ask) y aflojá a medida que ganás confianza en el comportamiento de Claude en tu proyecto. Las reglas deny para operaciones destructivas deberían estar siempre.
