🗂️ Scopes de Configuración Referencia

💡
Idea clave: Claude Code tiene cuatro niveles de configuración. Entender cuál aplica en cada contexto define qué se commitea al repo, qué es personal, y qué administra la organización.

📐 Los cuatro scopes

ScopeArchivo¿Se commitea?Quién lo controla
🏢 Managed Configuración central de la organización — IT / Administradores. Prevalece sobre todo. No editable por el usuario.
👤 User ~/.claude/settings.json ❌ No — es personal Vos. Aplica a todos tus proyectos. Preferencias personales, API keys, hooks globales.
📁 Project .claude/settings.json ✅ Sí — va al repo El equipo. Hooks, MCP, permisos y skills compartidos. Todo el equipo lo hereda al clonar.
🔒 Local .claude/settings.local.json ❌ No — Claude Code lo agrega a .gitignore automáticamente Vos, por proyecto. Tiene mayor prioridad que Project y User. Overrides locales: connection strings, tokens de desarrollo, prefs personales por repo.
⚠️
Nunca pongas secrets (tokens, passwords, connection strings reales) en .claude/settings.json — ese archivo va al repositorio. Usá siempre .claude/settings.local.json para valores sensibles.

🔁 Orden de precedencia

Cuando hay conflicto entre scopes, el orden de prevalencia (de mayor a menor) es:

Managed  >  Local  >  Project  >  User

Local sobreescribe Project y User — es el override personal por repositorio. Una regla deny en Managed no puede ser anulada por ningún otro scope.

📋 Qué va en cada scope — guía práctica

¿Qué configurar?Scope recomendado
Hooks de linting/build del proyecto📁 Project (.claude/settings.json)
MCP servers del proyecto (DB staging, GitHub)📁 Project (.claude/settings.json)
Connection string real de la base de datos local🔒 Local (.claude/settings.local.json)
Token personal de GitHub o API keys👤 User (~/.claude/settings.json)
Preferencias de modelo o formato de respuesta👤 User (~/.claude/settings.json)
Restricciones de seguridad para toda la organización🏢 Managed (administrado centralmente)
